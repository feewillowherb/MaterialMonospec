# 称重列表重复数据：问题分析与修复

**日期**：2026-07-13  
**状态**：已修复

---

## 1. 问题现象

- 称重结束后，左侧「未完成」列表偶发出现**两条看起来相同的记录**。
- **手动刷新后重复消失**。
- 在 ReactiveUI `MessageBus` 时期无此问题；改为 ABP `ILocalEventBus` 后出现。

界面控件 `WeighingRecordListView.axaml` 仅绑定：

```xml
<ItemsControl ItemsSource="{Binding ListItems}">
```

列表内容完全由 `AttendedWeighingViewModel.ListItems` 驱动，页面本身不会造数。

---

## 2. 根因

### 2.1 MessageBus → ILocalEventBus 的线程语义变化

| | ReactiveUI `MessageBus` | ABP `ILocalEventBus` |
|---|---|---|
| 回调线程 | 常与 UI / `ObserveOn(MainThread)` 配合 | 在**发布方线程**派发 |
| 本场景发布方 | 多为 UI 侧或已调度 | `WeighingRecordService` 在称重异步队列（线程池）`PublishAsync` |
| 迁移后缺口 | — | 列表刷新订阅**未**切回 UI 线程 |

OpenSpec `attended-weighing` 规范要求：

> 收到 ILocalEventBus 事件后，SHALL 通过 `ObserveOn(RxApp.MainThreadScheduler)` 确保在 UI 线程更新集合。

迁移时部分属性订阅做了 `Dispatcher.UIThread.Post`（如 `StatusChanged`），但 **`WeighingRecordCreated` / `MatchSucceeded` / `SaveCompleted` 等直接 `await RefreshAsync()`**，在后台线程改 `ObservableCollection`。

### 2.2 为何「刷新后消失」

1. 后台线程对 `ListItems` 做 `Clear` + 逐条 `Add`，Avalonia `ItemsControl` 易出现视觉重复/幽灵行。
2. 同时与搜索防抖、`InitializeOnFirstLoad` 等 UI 侧 `RefreshAsync` 交错时，可能出现 `Clear`/`Add` 竞态，集合短暂叠两份。
3. 用户手动刷新在 UI 线程完整重建列表 → 重复消失。

这与「库里落了两条、刷新仍在」的数据重复模式不符。

### 2.3 同文件内的对照（迁移遗漏）

**已切 UI 线程（正确）：**

- `StatusChangedEventData`
- `PlateNumberChangedEventData`
- `DeliveryTypeChangedEventData`

**迁移后未切 UI、会改列表（问题点）：**

- `WeighingRecordCreatedEventData` → `RefreshAsync`
- `MatchSucceededEventData` → `RefreshAsync` + 选中项
- `SaveCompletedEventData` → `RefreshAsync` + 选中项
- `UpdatePlateNumberEventData` → `RefreshAsync`
- `DetailOperationCompleted` / `DetailCloseRequested` / `ManualMatchSaveCompleted` → 导航与刷新

---

## 3. 错误归因澄清

曾怀疑 `AttendedWeighingService.OnWeightAndStatusChanged` 在状态转换与 backup 分支同一次入队两次 `OnWeightStabilizedAsync`。

该路径可能导致**真·双建库**，但：

- 现象「刷新后消失」说明主诉是 **UI/集合竞态**，不是持久化双行；
- 用户确认问题出现在 MessageBus 替换为 LocalEvent **之后**。

发射源加固可另开任务；本问题以 ViewModel 线程与刷新串行为准。

---

## 4. 修复方案（已落地）

### 4.1 `AttendedWeighingViewModel`

1. 增加 `RunOnUiAsync`，凡改 `ListItems` / 选中态的 LocalEvent 回调均 `Dispatcher.UIThread.InvokeAsync`。
2. `RefreshAsync` 使用 `SemaphoreSlim` 串行；分页属性与 `ListItems.Clear/Add` 在 UI 线程执行。

### 4.2 `UrbanAttendedWeighingViewModel`

`ReloadRecordsAsync` 同样串行，并用 `Dispatcher.UIThread.InvokeAsync` 更新集合（替换原先 fire-and-forget 的 `RxApp.MainThreadScheduler.Schedule`，避免并发 Clear/Add 交错）。

### 4.3 关键文件

| 文件 | 变更 |
|------|------|
| `repos/MaterialClient/src/MaterialClient.AttendedWeighing/ViewModels/AttendedWeighingViewModel.cs` | UI 调度 + `RefreshAsync` 串行 |
| `repos/MaterialClient/src/MaterialClient.Urban/ViewModels/UrbanAttendedWeighingViewModel.cs` | `ReloadRecordsAsync` 串行 + UI 调度 |

---

## 5. 验证方法

1. Recycle/标准称重：上磅稳定一次，未完成列表只新增一条。
2. 连续多次称重，列表无同 Id 叠行、无「幽灵双行」。
3. 称重过程中切换 Tab / 触发搜索防抖，不应出现短暂双份后需手动刷新才恢复。
4. 日志中 `CollectionChanged: ADD` 同一轮刷新内 Id 不重复。

---

## 6. 后续建议

- 新增 LocalEvent 订阅时：凡碰 Avalonia 绑定集合/选中态，默认 `Dispatcher.UIThread` 或 `ObserveOn(RxApp.MainThreadScheduler)`。
- Code review 对照：同一 ViewModel 内已有 `Post` 的订阅 vs 直接 `RefreshAsync` 的订阅是否遗漏。
- （可选）`AttendedWeighingService` backup 与状态转换同帧双入队：独立加固，防真·双建库。
