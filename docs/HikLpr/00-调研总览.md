# 海康 LPR 车牌解析异常调研总览

**调研日期**：2026-07-02  
**对比对象**：

| 项目 | 路径 | 状态 |
|------|------|------|
| GovClient（正常） | `Fdsoft.Weight.GovClient/BLL/CaptureDevice.cs` | 车牌解析正确 |
| MaterialClient（异常） | `repos/MaterialClient/.../HikvisionLprService.cs` | 车牌乱码、一次抓拍多条垃圾事件 |
| 异常日志 | `MaterialMonospec/_temp/log4.log` 342–364 行 | 见下文 |

---

## 结论摘要

**GovClient 能正常解析，不是因为用了某种特殊 GBK 技巧，而是因为：**

1. 使用 AlarmCSharpDemo 提供的 **与 HCNetSDK.h 一致** 的 P/Invoke 结构体（`CHCNetSDK.NET_ITS_PLATE_RESULT` 等）。
2. 从 **`struPlateInfo.sLicense`** 读取车牌（嵌套在 `NET_DVR_PLATE_INFO` 内），而非顶层虚构字段。
3. ITS 结果 **只处理一条** `struPlateInfo`，不遍历数组。
4. 运行在 **.NET Framework** 上，GBK 默认可用。

**MaterialClient 解析失败的主因是结构体布局错误（约占 90%），GBK 注册时序问题是次要因素（约占 10%）：**

| 问题 | 表现 | 严重度 |
|------|------|--------|
| `HikvisionSdk.cs` 中 `NET_ITS_PLATE_RESULT`、`NET_DVR_PLATE_RESULT`、`NET_DVR_ALARMER` 与官方 SDK 布局不一致 | 读到时间戳、UUID、设备名等非车牌字段；一次抓拍触发 ~11 条 LPR 事件 | **致命** |
| `HandleItsPlateResult` 将 `struPlateInfo` 当作 16 元素数组遍历 | 同一次抓拍连续发布多条乱码车牌 | **致命** |
| `HandlePlateResult` 使用顶层 `sLicense` 而非 `struPlateInfo.sLicense` | 旧版 COMM_UPLOAD_PLATE_RESULT 同样乱码 | **致命** |
| `NET_DVR_ALARMER` 布局错误 | 设备显示为 `Unknown (iDS-TCM204-E...&192.168.1.100)`，无法匹配配置 | **高** |
| 早期运行未注册 `CodePagesEncodingProvider` | 日志 114–120 行：`'GBK' is not a supported encoding name` | **中**（已部分修复） |

GBK 编码本身在 MaterialClient 的 `HikvisionEncodingHelper` 中实现正确；在结构体偏移正确的前提下，与 GovClient 的 `Encoding.GetEncoding("GBK")` 等价。

---

## 日志特征（log4.log 342–364）

一次抓拍在 **11ms 内** 连续产生约 **11 条** LPR 事件，车牌内容为：

- 乱码二进制片段
- `20260701165548354`（实为 `struSnapFirstPicTime` 时间戳格式 `yyyyMMddHHmmssfff`）
- `00000000000000000`（零填充字段）
- `b7d6e814babc66f47c28f505f`（UUID/会话 ID 片段）
- 空字符串、`3D` 等

设备名显示为：

```text
Unknown (iDS-TCM204-E 20250224AIFW6816131 ... IP CAPTURE CAMERA ... &192.168.1.100)
```

其中 `iDS-TCM204-E...` 来自 `NET_DVR_ALARMER.sDeviceName`，`192.168.1.100` 嵌在错误偏移的 IP 字段末尾——说明 **报警器结构体也已错位**。
