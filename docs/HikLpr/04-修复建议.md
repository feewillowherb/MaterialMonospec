# 修复建议与验收标准

## 1. 修复原则

1. **以 SDK 头文件为唯一真相**：从 `HCNetSDK.h`（CH-HCNetSDKV6.1.9.48）或 GovClient 的 `AlarmCSharpDemo/CHCNetSDK.cs` 移植结构体，不要手写简化结构。
2. **对齐 GovClient 的字段访问路径**：`struPlateInfo.sLicense`、`struPicInfo[]`、`dwPicNum`。
3. **保留 MaterialClient 已有优点**：`GCHandle` 固定回调、`NET_DVR_StopListen_V30`、`CodePagesEncodingInitializer`。

---

## 2. 必改项

### 2.1 重写 `HikvisionSdk.cs` 结构体

至少替换以下定义为与官方一致：

| 结构体 | 当前问题 |
|--------|----------|
| `NET_DVR_ALARMER` | 字段完全不对，导致 IP/设备名错乱 |
| `NET_DVR_PLATE_INFO` | 应使用官方 `NET_DVR_PLATE_INFO`，非自创 `NET_DVR_PLATE_INFO_EX` 替代 |
| `NET_DVR_PLATE_RESULT` | 缺少 `dwSize`、`struPlateInfo`、`pBuffer1` 等正确顺序 |
| `NET_ITS_PLATE_RESULT` | 误用 `dwResultNum` + `struPlateInfo[16]` 数组 |
| `NET_ITS_PICTURE_INFO` | 缺失，无法正确取图 |
| `NET_DVR_VEHICLE_INFO` | 缺失 |
| `NET_DVR_TIME_V30` | 缺失 |

可参考 GovClient 评估文档列出的字段清单：`struPicInfo[]`、`struSnapFirstPicTime`、`struPlateInfo`、`struVehicleInfo`、`dwPicNum`。

## 3. 建议项

| 项 | 说明 |
|----|------|
| 从 AlarmCSharpDemo 复制结构体 | 降低与 GovClient 行为漂移风险 |
| 增加结构体大小断言 | `Debug.Assert(Marshal.SizeOf<NET_ITS_PLATE_RESULT>() == expectedFromSdk)` |
| 单元测试 | 用 SDK 样例二进制或 GovClient 录制的 `pAlarmInfo` 字节做 golden test |
| 过滤无效车牌 | 复用 GovClient `Contains("车牌")` 与空串检查 |
| 日志脱敏 | 调试时可 Dump `pAlarmInfo` 前 256 字节十六进制，便于对比偏移 |

---

## 4. 验收标准

在 iDS-TCM204-E（192.168.1.100）上验证：

1. **一次抓拍只产生 1 条**有效 LPR 事件（允许因重传偶尔 2 条，不应有 10+ 条）。
2. 车牌为真实号牌（如 `浙Axxxxx`），无时间戳、UUID、长串 `0`。
3. 设备显示为配置中的名称，或 `sDeviceIP` 解析为 `192.168.1.100`（非 Unknown + 设备名拼接）。
4. 日志无 `'GBK' is not a supported encoding name`。
5. UrbanMode 下 LPR 附件 JPG 可正常保存（`struPicInfo` / `pBuffer1` 路径正确）。
6. 与 GovClient 对同一设备抓拍结果 **车牌字符串一致**。

---

## 5. 不建议的方向

- **仅调整 GBK 为 GB2312/UTF-8**：无法修复结构体错位，乱码依旧。
- **在乱码结果上做正则清洗**：掩盖根因，时间戳可能被误当成车牌。
- **增大 `struPlateInfo` 数组长度**：官方是单对象，不是数组。
