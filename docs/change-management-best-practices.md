# 变更管理最佳实践

## 变更创建

### 何时创建变更

需要创建变更的场景：
- 新功能开发
- 架构调整
- 跨仓库的改动
- 性能优化
- 破坏性更改

不需要创建变更的场景：
- 简单的 bug 修复
- 拼写错误修正
- 格式调整
- 非破坏性的依赖更新

### 变更命名规范

使用动词前缀 + 简短描述：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `add-` | 新功能 | `add-auth-module` |
| `update-` | 更新功能 | `update-weighing-flow` |
| `remove-` | 移除功能 | `remove-legacy-api` |
| `refactor-` | 重构 | `refactor-state-management` |
| `fix-` | 修复 | `fix-memory-leak` |

### 提案质量

好的 proposal.md 应包含：
1. **清晰的 Why**：解释为什么需要这个变更
2. **具体的 What Changes**：列出具体的变更内容
3. **准确的影响范围**：标明受影响的系统和模块

## 工作流建议

### 小型变更（1-2 天）
1. 创建变更 → 编写 proposal → 直接实现 → 归档
2. design.md 可简化

### 中型变更（3-7 天）
1. 创建变更 → proposal → design → specs → tasks → 实现 → 归档
2. 完整的工件链

### 大型变更（1 周以上）
1. 创建变更 → 完整工件 → 分阶段实施 → 逐步归档
2. 可能需要拆分为多个子变更

## 跨仓库变更

### 识别跨仓库影响

在创建提案时：
1. 检查变更是否涉及多个子仓库
2. 在 proposal.md 的 Impact 部分明确说明
3. 在 tasks.md 中按子仓库分列任务

### 协调策略

- 主仓库的 specs 变更在归档时自动提交
- 子仓库代码需各自手动提交
- 建议先完成主仓库文档变更，再实现子仓库代码

## 归档管理

### 归档前检查

- [ ] 所有 tasks.md 中的任务已标记完成
- [ ] specs 与代码实现一致
- [ ] 无遗留的 TODO 或临时文件
- [ ] 运行 `openspec validate <change-name> --strict` 通过

### 归档后

- 归档的变更保存在 `openspec/changes/archive/` 中
- 可随时查看历史变更作为参考
- 使用 `openspec list` 查看活动变更
