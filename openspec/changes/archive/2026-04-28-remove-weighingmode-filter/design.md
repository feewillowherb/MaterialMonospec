## Context

当前系统在 `Provider`、`Material`、`MaterialUnit` 的查询和选择流程中，将实体的 `WeighingMode` 字段作为启用过滤条件的一部分。该做法与字段当前定位不一致：`WeighingMode` 不再承担“是否启用”语义，并且计划在后续版本中删除。若继续依赖该字段，将导致行为歧义和迁移阻塞。

本次改动属于跨模块逻辑调整，涉及 `MaterialClient.Common` 的 API 契约语义、服务查询逻辑，以及上层选择控件的数据结果一致性。

## Goals / Non-Goals

**Goals:**
- 在不改变外部调用方式的前提下，移除 `Provider`、`Material`、`MaterialUnit` 基于 `WeighingMode` 的过滤。
- 统一三类实体的可见性语义：不再通过 `WeighingMode` 判定“可用/启用”。
- 为后续删除实体 `WeighingMode` 字段提供兼容过渡。

**Non-Goals:**
- 不在本次变更中物理删除数据库或模型中的 `WeighingMode` 字段。
- 不调整与 `WeighingMode` 仍有业务意义的其它模块（如业务单据模式判定）。
- 不引入新的启用字段或替代状态体系。

## Decisions

1. **保留接口结构，调整过滤语义**
   - 决策：保持现有 API 方法签名不变，仅在实现中去除 `WeighingMode` 相关 where 条件。
   - 理由：最小化调用方变更风险，降低回归范围。
   - 备选方案：修改 API 入参并清理全部调用点。未采纳原因是改动面过大，超出本次目标。

2. **按实体统一处理，不做局部特判**
   - 决策：`Provider`、`Material`、`MaterialUnit` 同步移除过滤。
   - 理由：避免出现实体间行为不一致，减少后续维护复杂度。
   - 备选方案：仅对单一实体先行移除。未采纳原因是会导致 UI 侧体验不一致。

3. **通过规格与任务明确“字段废弃但暂存”的阶段性策略**
   - 决策：在规格中明确该字段已不用于启用过滤，删除动作留待后续变更。
   - 理由：使当前变更范围可控，同时为下一阶段 schema 变更建立前置条件。

## Risks / Trade-offs

- **[风险] 结果集扩大导致用户看到更多历史数据** → **缓解**：在验收中覆盖列表/搜索场景，确认“全部可见”是预期行为。
- **[风险] 某些调用点仍有本地 `WeighingMode` 二次过滤** → **缓解**：全局搜索相关实体查询路径并统一清理。
- **[权衡] 保留旧字段会在短期内产生认知负担** → **缓解**：在实现注释和规格中明确字段废弃状态，并规划后续删除变更。
