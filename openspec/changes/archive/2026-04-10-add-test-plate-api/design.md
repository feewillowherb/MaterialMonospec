## Context

当前 Web 宿主 `MinimalWebHostService` 已承载地磅测试 API 与华夏智信回调 API，并直接访问共享 DI 容器发布 `LicensePlateRecognizedMessage`。现有联调依赖真实设备回调或主动抓拍触发，缺少一个无需设备参与、可重复调用的测试识别注入入口。  
本次变更需在保持现有回调兼容性的前提下，新增一个专用于测试联调的 API，最小化对现有识别流程（MessageBus 订阅侧）的改动。

## Goals / Non-Goals

**Goals:**
- 在 `MinimalWebHostService` 增加测试车牌注入端点，支持通过 HTTP 触发 `LicensePlateRecognizedMessage`。
- 强制 `plateNumber` 必填并进行 Trim 后非空校验。
- 允许可选字段为空，使用默认值补齐，保证消息结构完整且可追踪来源。
- 提供清晰响应与日志，便于自动化测试与现场联调排障。

**Non-Goals:**
- 不修改真实设备（华夏智信/Vzvision/海康）的回调协议与处理逻辑。
- 不引入新的持久化存储或历史记录表。
- 不调整消息消费侧业务规则（如匹配、锁定、门禁控制策略）。

## Decisions

1. **接口形态：新增独立 POST 路径**
   - 决策：新增例如 `/api/lpr/test-plate` 的独立接口，而非复用现有设备回调路径。
   - 原因：设备回调路径有既定字段协议（如 `type=online`），复用会混淆语义并增加兼容风险。
   - 备选方案：
     - 复用华夏智信回调接口并扩展参数：拒绝，协议污染且误触发风险高。
     - 在设置页直接调用内部服务：拒绝，无法满足外部联调脚本与 HTTP 自动化需求。

2. **请求模型：`plateNumber` 必填，其余可选**
   - 决策：请求体使用明确 DTO，`plateNumber` 做必填校验；`deviceType`、`deviceName`、`colorType`、`timestamp` 可选。
   - 原因：满足“车牌必填，其他可选”的需求，同时保持消息字段类型安全。
   - 备选方案：
     - 接收自由 JSON 字典：拒绝，运行期类型转换复杂，错误信息不稳定。

3. **消息组装默认策略**
   - 决策：当可选字段缺失时使用可预测默认值（如 `DeviceType` 默认 `Huaxiazhixin` 或约定的测试设备类型，`DeviceName` 默认 `TestApi`，`Timestamp` 默认 `DateTime.Now`）。
   - 原因：保证下游消费者在字段缺省情况下行为稳定，并可从来源字段区分测试事件。
   - 备选方案：
     - 缺失字段直接透传 null：部分字段是值类型/业务依赖字段，不利于兼容历史逻辑。

4. **错误处理与返回约定**
   - 决策：参数错误返回 400；内部异常返回 500；成功返回 200 且包含实际发送的关键字段。
   - 原因：与现有最小 API 风格一致，便于调用方区分“输入错误”和“系统错误”。

## Risks / Trade-offs

- **[测试接口被误用于生产流程]** → 通过接口命名、日志前缀、文档说明将其标注为测试用途；后续可增加配置开关限制。
- **[默认 `deviceType` 与消费侧规则耦合]** → 在规格中固定默认值并补充测试，避免未来变更造成行为漂移。
- **[非法车牌输入格式导致脏数据事件]** → 当前仅保证非空；若后续需要更严格格式校验，可基于 `PlateNumberValidator` 迭代增强（本次不强制）。
- **[新增 API 增加维护面]** → 与现有 `MinimalWebHostService` 同一文件实现，控制范围在单服务内，降低跨模块复杂度。

## Migration Plan

1. 在 `MinimalWebHostService` 增加新路由、请求 DTO 与参数校验逻辑。
2. 更新根路由 `endpoints` 列表，确保新接口可发现。
3. 增加单元/集成测试覆盖：必填校验、可选字段默认、消息发布成功路径。
4. 联调验证脚本切换到新接口，保留现有设备回调接口不变。
5. 回滚策略：若出现兼容问题，移除新增路由即可，不影响既有设备回调与地磅测试接口。

## Open Questions

- 是否需要加“仅测试模式可调用”的运行时开关（例如依赖设置项）以进一步降低误用风险？
