# Proposal: 更新软件设计文档 (SDD)

**Change ID**: `update-software-design-document`
**Created**: 2026-01-15
**Status**: Draft
**Priority**: P2

---

## 概述

### 背景

MaterialClient 是一个基于 C# 13 / .NET 10.0 和 Avalonia UI 11.3.9 的 Windows 桌面应用程序,采用 MVVM 模式与 ReactiveUI 进行状态管理。项目经历了多次架构演进和技术优化,但现有的软件设计文档(Software Design Document, SDD)未能及时更新以反映这些变化。

### 问题陈述

当前项目的 SDD 存在以下主要问题：

1. **文档过时**: 技术栈版本、架构模式、核心设计决策未及时更新
2. **缺少关键设计决策**: RxState 状态管理模式、硬件抽象层、内存泄漏防护策略等未系统化记录
3. **架构视图不完整**: 缺少组件图、时序图、数据流图等架构可视化内容
4. **技术债务追踪缺失**: 已知问题和优化建议未集中记录

### 拟议方案

通过两个阶段的系统性更新，创建并维护一份全面、准确、易维护的 SDD：

**阶段一 - 评估**: 分析现有文档和代码库,识别差距和质量问题
**阶段二 - 更新**: 补充和更新 SDD 的核心章节、架构视图、技术决策记录和开发指南

## 范围

### 范围内

#### 核心章节更新

- **架构概述**: 技术栈版本(.NET 10.0, Avalonia UI 11.3.9, EF Core 10.0.1)
- **模块设计**: 各模块职责边界和接口定义
  - `AttendedWeighingService` - 有人值守称重服务
  - `UnattendedWeighingService` - 无人值守称重服务
  - `TruckScaleWeightService` - 地磅重量服务
  - `HikvisionCameraService` - 海康摄像头服务
  - `PlateRecognitionService` - 车牌识别服务
  - `ApiSyncService` - 远程平台同步服务
- **状态管理架构**: RxState 模式实现 (State, Reducer, Action, Side-effect)
- **数据模型**: 核心实体设计 (`WeighingRecord`, `Waybill`, `Material`, `Provider`)

#### 架构视图补充

- **组件图**: UI 层、Service 层、Hardware 层、Data 层的关系
- **时序图**: 关键业务流程
  - 有人值守称重流程
  - 自动匹配流程
  - 远程同步流程
- **数据流图**: Rx 管道的数据流向和状态转换
- **部署图**: 单机部署的组件依赖关系

#### 技术决策记录

- **选择 Rx.NET 的原因**: 响应式状态管理 vs 传统事件驱动
- **硬件抽象策略**: 接口隔离 + Mock 实现用于测试
- **内存泄漏防护**: Subscription disposal 规范、RefCount 使用、Buffer 大小限制
- **数据库选型**: SQLite vs 其他方案(单用户桌面应用场景)

#### 约束和风险

- **平台约束**: Windows x64 only、.NET 10.0 runtime、HCNetSDK 依赖
- **硬件约束**: 串口独占性、摄像头带宽限制、网络依赖性
- **性能约束**: 24/7 运行要求、高频率重量流处理
- **已知问题**: 当前待优化的技术债务列表

#### 开发指南

- 集成现有开发规范
- 硬件集成最佳实践
- Rx 编程规范和内存泄漏防护指南
- 测试策略 (Unit, Integration, Memory Leak)

### 范围外

- 代码重构或优化（仅文档更新）
- 新功能开发
- 性能测试和优化实施
- 第三方集成细节的深入文档(仅记录接口和约定)

## 影响分析

### 预期收益

1. **提升可维护性**: 新开发者能快速理解架构和设计决策,减少 onboarding 时间
2. **辅助 AI 开发**: 完整的 SDD 为 AI 助手提供准确的上下文,减少理解偏差和代码生成错误
3. **降低知识流失**: 关键设计决策有据可查,减少对核心开发者的依赖
4. **规范开发流程**: 明确的架构边界和编码约定,提升代码质量和一致性

### 工作量估算

| 阶段 | 任务 | 预计工作量 |
|------|------|-----------|
| 阶段一 | 评估当前 SDD 状态 | 4-6 小时 |
| 阶段二 | 核心 SDD 章节更新 | 8-12 小时 |
| 阶段二 | 架构图绘制 | 4-6 小时 |
| 阶段二 | 技术决策记录编写 | 3-4 小时 |
| 阶段二 | 开发指南编写 | 2-3 小时 |
| 评审和修订 | 团队评审和迭代 | 4-6 小时 |
| **总计** | | **25-37 小时** |

### 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 文档可能随代码演进再次过时 | 高 | 中 | 在 OpenSpec workflow 中增加 SDD 更新检查项,定期(如每季度)审查 |
| 架构图维护成本高 | 中 | 中 | 使用 Mermaid 等文本绘图工具,便于版本控制和协作编辑 |
| 技术细节过多导致文档臃肿 | 中 | 低 | 分层级组织文档(概览 + 专题深入),保持高可读性 |
| 描述不准确或与实现不一致 | 高 | 中 | 与团队评审确认,代码变更时同步更新相关章节 |

### 依赖

- 需要团队确认架构描述的准确性
- 需要访问现有分析报告(如 `docs/AttendedWeighingService-RxState-Optimization-Report.md`)
- 需要与现有规格文档(`specs/`)保持一致性

### 成功标准

1. **完整性**：SDD 包含所有核心章节（架构、模块、数据模型、状态管理、技术决策、约束、开发指南）
2. **准确性**: 技术栈版本、模块职责、设计决策与当前实现一致
3. **可读性**: 架构图清晰,章节结构合理,易于导航和查找
4. **可维护性**: 使用文本格式(Mermaid)绘制图表,便于版本控制和协作
5. **评审通过**: 团队评审确认内容准确,无明显遗漏或错误

## 已考虑替代方案

### 替代 1：仅更新过时信息

**描述**：仅更新技术栈版本和基本信息，不补充架构视图和设计决策

**优点**: 工作量小,快速完成
**缺点**: 未解决核心问题(架构视图缺失、设计决策未记录),价值有限

**选择理由**: 当前项目架构复杂度高,需要系统化的文档支持,简单更新不足以满足需求

### 替代 2：使用自动化工具生成文档

**描述**：使用代码分析工具（如 DocFX、StyleCop）自动生成 API 文档

**优点**: 自动化,与代码同步
**缺点**: 仅能生成 API 文档,无法捕获架构设计、决策背景和最佳实践

**选择理由**: 自动化工具可作为补充,但无法替代人工编写的高质量 SDD

### 替代 3：维持现状 + 内部 Wiki

**描述**：不创建正式 SDD，使用内部 Wiki 或 Notion 记录架构信息

**优点**: 灵活,易于编辑
**缺点**: 与代码库分离,版本控制差,无法为 AI 提供准确上下文

**选择理由**: SDD 应与代码库一起版本控制,确保一致性和可追溯性

## 实施方式

### 阶段 1：评估（阶段一 - 评估）

1. **定位现有文档**
   - 检查 `docs/` 目录下的现有设计文档
   - 识别与 SDD 相关的分析报告
   - 查看 `specs/` 目录下的规格文档作为参考

2. **差距分析**
   - 对比现有文档与项目实际实现
   - 识别缺失的设计决策和架构视图
   - 列出需要补充的关键技术点

3. **评估文档质量**
   - 检查文档的完整性、准确性和可维护性
   - 识别过时的技术栈版本和依赖项
   - 评估文档对新人理解和 AI 辅助开发的友好度

### 阶段 2：更新（阶段二 - 更新）

1. **核心章节更新**
   - 更新架构概述为当前技术栈版本
   - 详细描述各模块职责和接口
   - 记录 RxState 模式的实现
   - 更新数据模型定义

2. **架构视图补充**
   - 使用 Mermaid 绘制组件图
   - 使用 Mermaid 时序图描述关键业务流程
   - 使用 Mermaid 流图描述 Rx 数据流
   - 绘制部署图

3. **技术决策记录**
   - 记录 Rx.NET 选型原因
   - 记录硬件抽象策略
   - 记录内存泄漏防护策略
   - 记录数据库选型原因

4. **约束和风险**
   - 记录平台约束
   - 记录硬件约束
   - 记录性能约束
   - 列出已知技术债务

5. **开发指南**
   - 集成现有开发规范
   - 添加硬件集成最佳实践
   - 编写 Rx 编程规范
   - 定义测试策略

### 阶段 3：评审与迭代（评审和迭代）

1. **团队评审**: 与团队确认架构描述的准确性
2. **修订完善**: 根据反馈修订文档
3. **建立维护机制**: 定义 SDD 与代码变更的同步机制

## Maintenance Strategy

### Documentation Maintenance

1. **定期审查**: 每季度审查 SDD,更新过时信息
2. **变更触发**: 重大架构变更时更新相关章节
3. **OpenSpec 集成**: 在 OpenSpec workflow 中增加 SDD 更新检查项
4. **版本控制**: SDD 与代码库一起版本控制,确保可追溯性

### Diagram Maintenance

- 使用 Mermaid 等文本格式,便于版本控制和协作编辑
- 架构图与代码实现保持同步
- 重大架构变更时更新相关图表

## Open Questions

1. SDD 应该采用单一文档(`docs/SDD.md`)还是多文档结构(`docs/sdd/` 目录)?
   - 建议: 单一文档用于概览,专题文档用于深入

2. 是否需要为每个模块单独创建详细设计文档?
   - 建议: 先完成核心 SDD,根据需求补充专题文档

3. SDD 的更新频率如何定义?
   - 建议: 每季度定期审查 + 重大变更时更新
