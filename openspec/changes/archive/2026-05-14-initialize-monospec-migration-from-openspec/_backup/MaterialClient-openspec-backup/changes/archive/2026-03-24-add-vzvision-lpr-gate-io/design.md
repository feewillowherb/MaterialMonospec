## Context

当前车牌识别配置页面 `AddLprDialog` 已按设备类型动态展示字段，但尚无道闸 I/O 联动配置。  
同时，`VzvisionSdk` 已具备 `VzLPRClient_SetIOOutputAutoResp(handle, ioChannel, 500)` 可调用能力，说明底层接口可用，但业务链路缺少“识别后自动触发”的统一封装。  
本次变更要求将该能力定义为 LPR 通用可选能力，但当前仅在 Vzvision 场景落地；对其他设备类型通过能力门控与日志提示处理，并且避免把设备 I/O 控制直接塞入识别服务，保持职责分离与后续扩展性。

## Goals / Non-Goals

**Goals:**
- 在 LPR 配置中新增可持久化字段：`EnableGateIo` 与 `IoChannel`。
- 在收到识别结果后，若配置启用且设备类型为 Vzvision，则向对应设备发送 `VzLPRClient_SetIOOutputAutoResp(..., 500)` 开闸脉冲（500ms 自动复位）。
- 建立职责分离：识别服务与 I/O 控制服务解耦，通过清晰接口/编排连接。
- 明确非 Vzvision 设备当前不支持该行为，跳过执行并打印“设备未支持道闸 I/O”日志。

**Non-Goals:**
- 不实现其他厂商（如 Hikvision、Huaxiazhixin）的道闸 I/O 能力。
- 不引入复杂开闸策略（脉冲时长、重复触发去重策略、多级联动）。
- 不在本提案中改造 UI 交互视觉风格，仅新增必要配置项。

## Decisions

1. **在配置模型增加通用字段，但按设备能力控制可见与生效范围**
   - 选择：配置层新增 `EnableGateIo`、`IoChannel`，UI 仅在 Vzvision 时展示并可编辑。
   - 原因：字段可统一持久化，避免未来再做数据迁移；同时通过设备类型约束保证能力边界。
   - 备选：仅在 Vzvision 专有配置对象增加字段。放弃原因：模型碎片化，后续多厂商支持时扩展成本更高。

2. **识别与 I/O 下发通过 MessageBus + 独立服务编排**
   - 选择：识别服务发布识别消息（或专用 I/O 触发消息）；独立 `GateIoService`（命名可调整）订阅 `MessageBus` 后判断配置并调用 `VzvisionSdk.SetIOOutput`。
   - 原因：复用现有 MessageBus 架构，满足职责分离，降低识别服务与硬件控制耦合，便于独立维护与扩展。
   - 备选：在 `VzvisionLprService` 内直接调用 SDK。放弃原因：职责混杂，后续维护与测试复杂。

3. **发送语义固定为“500ms 自动复位开闸脉冲”**
   - 选择：收到有效识别后发送 `SetIOOutputAutoResp(ioChannel, 500ms)` 自动复位开闸信号。
   - 原因：带响应确认且满足 SDK 最小脉冲时长约束，满足当前需求最小闭环。
   - 备选：可配置输出值。放弃原因：当前无业务需求，增加错误配置风险。

4. **显式供应商能力门控**
   - 选择：能力配置对所有 LPR 类型可建模，但只有 `LprDeviceType = Vzvision` 且 `EnableGateIo = true` 时才触发 I/O；其余类型写入未支持日志并跳过。
   - 原因：既满足“当前仅 Vzvision 支持”，也为未来扩展其他厂商预留统一配置模型。
   - 备选：非 Vzvision 场景静默跳过。放弃原因：缺少可观测性，不利于现场排障和能力宣导。

## Risks / Trade-offs

- **风险：识别事件重复导致短时间多次开闸** -> 先按“每次识别触发一次”实现，并在日志中记录触发来源；后续按现场反馈决定是否加节流。
- **风险：ioChannel 配置错误导致触发失败** -> 在 UI 增加基础校验（非负/范围），并在触发失败时记录明确错误日志。
- **风险：用户在非 Vzvision 设备上启用配置后误以为已生效** -> 在识别触发路径输出“当前设备未支持道闸 I/O”日志，必要时增加 UI 提示。
- **风险：MessageBus 订阅管理不当导致重复触发** -> 将 I/O 订阅生命周期与服务生命周期绑定，确保单实例订阅并在停止时释放订阅。
- **权衡：先仅支持输出值 1，能力较窄** -> 以最小可用实现上线，后续如需脉冲宽度再扩展配置项。
