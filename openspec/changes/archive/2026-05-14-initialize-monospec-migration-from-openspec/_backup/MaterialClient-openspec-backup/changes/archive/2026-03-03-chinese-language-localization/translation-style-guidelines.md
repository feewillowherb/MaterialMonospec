# 翻译风格指南

本文为 MaterialClient 项目中技术文档与代码注释的翻译提供风格指南。

## 总体原则

### 准确优先
- 翻译意图与含义，而非仅字面
- 优先保持技术准确性而非逐字翻译
- 术语不确定时查阅技术参考
- 确保代码注释准确描述代码行为

### 一致性
- 所有译文中术语一致
- 标准术语遵循翻译词汇表
- 保持格式与结构一致
- 同类内容使用相同风格

### 清晰
- 使用清晰自然的中文
- 避免过于复杂的句式
- 尽量简洁直接
- 确保译文易于理解

## 技术文档指南

### Markdown 文档结构
- 保留所有 Markdown 格式（标题、列表、代码块、链接等）
- 保持原文档层级
- 代码块保持原样（块内英文注释可翻译）
- 保留所有链接与引用

### 标题
- 将标题内容译为中文
- 保持标题级别（H1、H2、H3 等）
- 标题格式一致
- 用作标题的技术缩写不翻译（如 API、HTTP、JSON）

### 列表
- 将列表项译为中文
- 保持列表类型（有序、无序、嵌套）
- 保留缩进
- 列表标记一致

### 代码块
- 可翻译代码块内的注释
- 不翻译代码本身（变量、方法、类等）
- 保留代码格式与缩进
- 错误信息与日志输出保持原语言

### 链接与引用
- 链接文字译为中文
- 保留所有 URL 与文件路径
- 内部引用链接保持可用
- 保持外部链接目标

### 表格
- 表格内容译为中文
- 保留表格结构与格式
- 表头一致
- 保持列对齐

### 强调
- 保留粗体、斜体等强调格式
- 保持相同的强调点
- 不无故增加或删除强调

## 代码注释指南

### 注释目的
- 代码注释应说明代码为何存在，而非在做什么
- 描述代码的目的与上下文
- 解释不直观的逻辑或算法
- 记录设计决策与权衡

### 注释风格
- 使用清晰自然的中文
- 语言简洁
- 随代码变更及时更新注释
- 避免仅复述代码的冗余注释

### 注释位置
- 注释放在所描述代码之前
- 注释紧贴相关代码
- 少用行内注释
- 将相关注释分组

### 特殊注释类型

#### TODO 注释
```csharp
// TODO: 实现新的称重模式处理逻辑
```

#### FIXME 注释
```csharp
// FIXME: 修复车牌识别超时问题
```

#### HACK 注释
```csharp
// HACK: 临时解决方案，等待硬件更新后移除
```

#### NOTE 注释
```csharp
// NOTE: 这个方法需要异步执行以避免阻塞 UI
```

#### WARNING 注释
```csharp
// WARNING: 不要在 UI 线程调用此方法
```

## 技术术语指南

### 缩写与简称
- 常用技术缩写保留英文：API、HTTP、REST、JSON、XML 等
- 较不常见术语使用完整中文译名并在括号中附英文
- 缩写用法保持一致

### 品牌与产品名
- 品牌名保留英文：Microsoft、Windows、Avalonia 等
- 产品名保留英文：Entity Framework、.NET 等
- 视受众需要可使用中文译名

### 代码元素
- 不翻译：变量名、方法名、类名、属性名等
- 翻译描述代码元素的注释
- 用中文描述代码在做什么

### 文件名与路径
- 文件名保留英文（面向用户的除外）
- 目录名保留英文
- 翻译对文件路径的描述
- 完整路径保持原样

## 格式指南

### 标点
- 中文内容使用中文标点（全角）
- 代码与技术内容使用英文标点
- 标点用法一致
- 中英文之间酌情加空格

### 数字与日期
- 使用西式数字（0-9）
- 日期格式一致（建议 YYYY-MM-DD）
- 数字格式符合受众
- 版本号保持标准格式

### 空白
- 保留原有换行
- 保持段落间距
- 缩进一致
- 标点周围空格适当

## 本地化考虑

### 日期与时间
- 时间使用 24 小时制
- 日期按中文习惯（年-月-日）
- 显示时间戳时考虑时区
- 适当使用相对时间表达

### 数字与单位
- 适当使用公制单位
- 数字格式符合中文习惯
- 涉及计量时注明单位
- 小数分隔符一致

### 地址与联系信息
- 地址按中文习惯排版
- 国家名使用正式名称
- 电话号码格式适当
- 邮政编码保持标准格式

## 质量保证指南

### 审校流程
- 审校所有译文的准确性
- 对照词汇表检查一致性
- 确认技术术语处理正确
- 确保格式保留

### 测试
- 测试已译文档的可读性
- 验证代码注释准确
- 检查链接与引用有效
- 确保 UI 元素正确显示

### 常见问题避免
- 机翻产生不自然中文
- 翻译应保留英文的代码元素
- 术语使用不一致
- 技术准确性丢失
- 破坏文档结构的格式错误

## 工具与资源

### 推荐工具
- 复杂技术内容由专业人工翻译
- 追求速度时可用 AI 翻译并人工审校
- 使用术语管理系统保证一致性
- 使用翻译记忆工具复用

## 示例

### 良好翻译
```csharp
/// <summary>
/// 获取指定车牌号的所有称重记录
/// </summary>
/// <param name="plateNumber">车牌号</param>
/// <returns>称重记录列表</returns>
public async Task<List<WeighingRecord>> GetRecordsByPlateNumber(string plateNumber)
{
    return await _context.WeighingRecords
        .Where(r => r.PlateNumber == plateNumber)
        .ToListAsync();
}
```

### 不佳翻译
```csharp
/// <summary>
/// 获取指定车牌号的全部称重记录
/// </summary>
/// <param name="plateNumber">车牌号码</param>
/// <returns>称重记录的列表</returns>
public async Task<List<WeighingRecord>> GetRecordsByPlateNumber(string plateNumber)
{
    return await _context.WeighingRecords
        .Where(r => r.PlateNumber == plateNumber)
        .ToListAsync();
}
```

**说明**：
- 「所有」与「全部」均可，保持一致即可
- 「车牌号」较「车牌号码」更简洁常用
- 「称重记录列表」较「称重记录的列表」更简洁

## 结论

遵循本指南可确保 MaterialClient 项目中的翻译：
- 准确且技术正确
- 在文档与代码中一致
- 清晰易懂
- 专业且格式规范

定期审阅与更新本指南有助于在项目演进中保持翻译质量。
