## Context

本次变更横跨 `MaterialClient.UI`（系统设置）、`MaterialClient.Urban`（称重列表与审批交互）和 `MaterialClient.Common`（DTO/服务调用约束）三个层面，属于跨模块行为增强。当前异常阈值仅在配置文件中可改，且审批链路缺少异常原因、上传时间、二次确认、称重日期上下文，影响现场审核效率与准确性。

约束条件：
- 仅 Urban 模式展示与生效异常阈值设置，避免影响标准/固废模式。
- 保持现有审批后“重算异常 + 复位同步状态”的核心业务语义。
- UI 优先以现有 ViewModel/DTO 模式扩展，避免引入并行状态源。

## Goals / Non-Goals

**Goals:**
- 在系统设置页最下方增加 Urban 异常阈值配置区域，并在 Urban 模式显示。
- 在 Urban 左侧列表新增“异常原因”“上传时间”字段，提升审核可读性。
- 将审批入口限制为“仅异常可点”，并在提交审批前强制二次确认。
- 在 `WeighingRecordEditDialog` 增加只读称重日期字段。

**Non-Goals:**
- 不改动 UrbanManagement 服务端审批接口契约。
- 不改动异常判定公式本身（仍使用上/下限 + 偏差百分比）。
- 不引入新的审核工作流状态机（仅增强现有交互）。

## Decisions

1. **设置项承载方式：沿用现有 SettingsEntity/SystemSettings 持久化通道**
   - 方案：将 `UrbanAnomalyDetection` 的三个阈值作为系统设置可编辑字段，保存时复用 `ISettingsService.SaveSettingsAsync`。
   - 原因：最小化数据流改造，保持设置页与业务层读取口径一致。
   - 备选：单独配置文件热写入；被拒绝（部署维护复杂、与现有设置体验割裂）。

2. **Urban 模式可见性：由 ViewModel 基于 WeighingMode 判定**
   - 方案：`SettingsWindowViewModel` 在加载时读取当前模式，仅当 `UrbanMode` 时显示阈值区块。
   - 原因：UI 行为可测试、避免 XAML 层硬编码业务判断。
   - 备选：XAML 直接绑定多值转换；被拒绝（可维护性差）。

3. **异常原因来源：由列表 DTO 直接提供“可展示文本”**
   - 方案：扩展 `UrbanWeighingListItemDto` 新增 `AnomalyReason` 与 `UploadTime`（可空）字段，UI 直接渲染。
   - 约束：`AnomalyReason` 使用短文案，优先 4~8 字短语（如“车牌为空”“超上限”“低下限”）。
   - 原因：避免 UI 侧重复推导业务规则，降低前端/后端解释不一致风险。
   - 备选：UI 根据记录字段自行推导原因；被拒绝（规则重复、易偏差）。

4. **审批可点击规则：命令可执行条件 + 视觉禁用双保障**
   - 方案：`ApproveRecordCommand` 在 `IsAnomaly=false` 时不可执行，按钮同步禁用样式。
   - 原因：同时防止误触和绕过。
   - 备选：仅在点击时弹提示；被拒绝（交互噪声高，且误触成本高）。

5. **二次确认触发点：在编辑结果校验通过后、持久化前**
   - 方案：用户点击“确定”后先做本地格式校验，通过后弹确认框，再调用更新服务。
   - 原因：避免无效输入也弹确认，减少多余交互。
   - 备选：进入对话框即要求确认；被拒绝（流程过重）。

## Risks / Trade-offs

- **[风险] 异常原因文案与实际判定不一致** → 由同一判定服务/映射层生成展示文案，禁止 UI 二次推导。  
- **[风险] 旧数据无上传时间导致显示空白** → `UploadTime` 允许为空，UI 显示 `--`。  
- **[风险] 新增设置仅 Urban 可见导致用户误解“配置丢失”** → 在非 Urban 模式下完全隐藏，不清空已存值。  
- **[风险] 审批按钮禁用可能影响既有手工修正流程** → 通过异常原因与筛选入口明确“仅异常审批”的产品意图。  

## Migration Plan

1. 扩展设置与列表 DTO 字段（向后兼容可空字段）。  
2. 更新 SettingsWindow 与 Urban 列表/审批 UI。  
3. 增加审批确认逻辑与只读称重日期展示。  
4. 回归验证：Urban 与非 Urban 模式显示差异、审批可点击条件、确认框流程、数据保存行为。  
5. 若需回滚，可仅回退前端展示与命令启用逻辑，不影响已有记录数据结构。  

## Open Questions

- 异常原因是否需要“多原因并列”展示（如“车牌为空；重量超上限”）还是仅首要原因？  
- 上传时间字段应展示“上云时间（客户端）”还是“服务端接收时间”，最终以哪一端为准？
